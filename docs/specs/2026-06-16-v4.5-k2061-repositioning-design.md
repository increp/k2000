# v4.5(C) — K2061 engine re-positioning

**Status:** Design proposed, 2026-06-16.

**Scope:** A *direction-setting* design — no engine code this phase. It re-targets the engine's north star from the **K2000** to the **K2061/K2088** VAST architecture, locks the unified engine model (a constant Summit voice bracketing a flexible K2061 source/DSP graph), establishes a GUI-grows-with-the-engine principle, and **rewrites the v5+ roadmap** around it. Each re-framed phase gets its own spec/plan later.

**Context:** v4.5 is three independent sub-projects: **A** — fix the Pirkle synth-book OCR in `k2000-kb` (now narrow: the K2088 manual ingested cleanly, so only the one garbled PDF remains); **B** — a Summit-look GUI, reframed below as the *UI foundation*; **C** — this re-positioning. The "k2000" repo name stays a codename (no rename).

**Sources:** K2088 Musician's Guide (pp. 48–50, 98–103) and the Novation Summit User Guide, via the `k2000-kb` reference KB.

## Thesis

**A Summit you can't break, with a K2061 VAST graph for unlimited sound generation underneath.**

The project's vision swaps its *flexibility* reference from K2000 → **K2061/K2088 VAST**, while the *surface and analog voice* are 100% **Summit**. The two compose as a hard-bracketed hybrid: everything that **generates** sound is flexible K2061 VAST; everything that **shapes and modulates** the voice is a constant Summit. You can never dial up a dead control or a patch that isn't a "real synth."

## The unified engine model (locked)

```
  K2061 Dynamic VAST — source + DSP graph   (VARIABLE)
  ┌────────────────────────────────────────────────────────┐
  │  source blocks:  [ Summit osc (default) | KVA | FM |     │
  │                    wavetable | sample/keymap | noise ]   │
  │  + DSP blocks:   [ mixer | ring mod | shaper | EQ | … ]  │
  │  wired serial / parallel  →  one voice output            │
  └───────────────────────────┬────────────────────────────┘
                              ▼
  Summit analog spine  (CONSTANT — always present, always live)
  ┌────────────────────────────────────────────────────────┐
  │      Huggett multimode filter + drive  →  VCA  →  out    │
  └────────────────────────────────────────────────────────┘
        ▲ driven by the CONSTANT Summit modulation system:
          envelopes (amp + mod) · LFOs · mod matrix · voice modes
          (always present; can modulate the spine AND any graph param)
```

The boundary is crisp:

- **Variable (K2061 Dynamic VAST):** every sound *source* — including the Summit oscillators, which are subsumed as just one (default) source block — plus DSP plumbing blocks, wired into arbitrary serial/parallel graphs producing one voice output. This is where "pure FM," "no filter," "weird DSP plumbing" lives.
- **Constant (Summit spine + modulation):** the Huggett multimode filter + drive → VCA, and the whole Summit modulation system — amp/mod envelopes, LFOs, the mod matrix, and voice modes (mono/poly/unison/glide). Always present, always live; touch the filter and the sound changes regardless of source. The mod matrix and envelopes/LFOs can modulate both the constant spine *and* any graph parameter.

This is VAST's own "pitch always first, amplitude always last" principle, extended so that the **Summit analog character is always the final shaping stage**.

## K2088 architecture reference (sourced)

Hierarchy: **Multi** (≤16 programs) → **Program** (≤32 layers, 256 voices) → **Layer** (source → algorithm → layer-insert FX) → **Algorithm** (DSP functions; fixed factory *or* Dynamic-VAST user graph).

- **Dynamic VAST** (Wiring Alg Editor): user algorithms where each function block has configurable **inputs / outputs / block size**, wired into serial **+ parallel** graphs with splits and joins ("thousands of wiring schemes"). Factory algorithms are preset graphs.
- **KVA oscillators** (anti-aliased synth oscillators, generated at the algorithm stage); **FM layers** (6-operator FM); **Cascade Mode** (route a layer's audio into another layer's algorithm); **KDFX** (per-layer insert FX chain + common insert + two aux chains); **CC sequencer**, **FUNs**, intonation/tuning maps, KSR.

## Current → target gap

| Subsystem | Now (v4) | K2061/K2088 target |
|---|---|---|
| Algorithm / routing | fixed linear list (4 algos, palette filter+shaper) | **Dynamic VAST** user graphs: variable-I/O blocks, serial **+ parallel**, splits/joins |
| Sources | 1 basic waveform osc (in the algorithm) | source **blocks**: Summit osc (default) + **KVA** + **FM** + wavetable + sample/keymap + noise |
| Analog back-end | filter is an optional palette block; basic amp | **constant Summit spine**: Huggett filter + drive → VCA (filter promoted *out* of the optional graph) |
| Modulation | per-block params only | constant Summit system: envelopes, LFOs, **mod matrix**, voice modes, CC seq, FUNs |
| Layers / voices | 2 layers, 64-voice pool | up to **32 layers**, **256 voices** |
| Effects | none | **KDFX** — per-layer insert + common insert + two aux chains |
| Inter-layer | independent, summed | **Cascade Mode** |
| Multitimbral | single program | **Multis** (≤16 programs) |
| Tuning | equal | intonation + tuning maps, KSR — **deferred to v11** |

The v1–v4 foundation (Program/Layer/Algorithm/palette, per-layer routing, cumulative preset migration) carries forward intact. The v3 fixed `AlgorithmLibrary` becomes **"factory algorithm" presets** — the floor; Dynamic VAST graphs are the ceiling. The SVF filter is **promoted out** of the optional palette into the constant spine.

## GUI principle: the UI grows with the engine

A Dynamic-VAST graph engine with 32 layers, KDFX, and a mod matrix is unusable through a primitive knob panel — so **GUI is a continuous, first-class concern, not a late phase**:

1. **B (v4.5) is the UI *foundation*, not a reskin:** an extensible Summit-aesthetic component/panel framework with theming and tiered-immediacy navigation. Load-bearing — everything after plugs into it, so it lands **before** v5.
2. **Every engine phase ships its own feature UI** on that foundation — a phase isn't done until you can *drive* it (v5 ships the graph editor, v8 ships the FX UI, etc.).
3. **The panel is permanently Summit-shaped for the constant spine** (filter, envelopes, LFOs, mod, amp in fixed places, always live), while the **source/DSP region is dynamic** — the active block's knob-cluster swaps in (3-osc block → osc knobs; FM block → operator knobs).
4. **v12+ shrinks** to final photoreal art + the long-tail polish — not "first real UI."

## Open questions (resolve at the noted phase)

- **Summit FX vs KDFX** *(v8):* KDFX is the routing architecture (per-layer insert + common + 2 aux); Summit's effects are effect *types* placed in those chains. Open: does a Summit-style master FX section stay a *constant* (always-present common chain, like the spine) or is it just one KDFX configuration?
- **Dynamic control panels** *(B + per-block phases):* how much of the variable source/DSP region lives on the front panel vs. pages; how the graph editor surfaces block params; how knob-clusters swap without losing state.
- **Source / mixer placement** *(v6/v7):* the mixer (osc/ring/noise levels) is graph-side now that sources are blocks — confirm it's a DSP block, and how multiple sources sum into the single voice output the spine expects.
- **Mod matrix scope** *(v5/v7):* the constant mod matrix must address both constant-spine params and arbitrary graph-block params — needs a stable parameter-addressing scheme as graphs change.
- **Voice/layer scale** *(v9):* allocation strategy at 32 layers / 256 voices; the v4 shared-pool generalizes, but stealing/limits need design.
