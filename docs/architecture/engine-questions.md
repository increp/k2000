# Engine architecture — decisions & open questions (living register)

A living register of architectural decisions and open questions for the K2061/Summit engine. Per `feedback_question_heavy_design`: questions are raised constantly, recorded here with the phase that resolves them, and **groomed for internal consistency before any ADR / spec / roadmap is written**. Resolved entries keep the answer so the rationale survives.

Source of the model: [v4.5(C) re-positioning spec](../specs/2026-06-16-v4.5-k2061-repositioning-design.md).

## Locked decisions

| # | Decision | Where |
|---|---|---|
| L1 | Engine north star is **K2061/K2088 VAST**, not K2000. Surface + analog voice stay **Summit**. | C spec |
| L2 | **Constant Summit spine**: a **selectable, live-switchable filter model** (Huggett default) + drive → VCA, plus the modulation system (amp/mod envelopes, LFOs, mod matrix, voice modes), always present. | C spec + v5 |
| L3 | **Sources are VAST blocks** — Summit oscillators are subsumed as one (default) source block; KVA/FM/wavetable/sample/noise are others. | C spec |
| L4 | **Dynamic VAST graph** (variable-I/O blocks, serial + parallel) is the variable source/DSP stage feeding the spine. v3's fixed `AlgorithmLibrary` = "factory" presets (the floor). | C spec |
| L5 | **GUI grows with the engine** — B (v4.5) is the load-bearing UI foundation (before v5); each phase ships its feature UI; constant spine = permanent panel, source/DSP region = dynamic knob-clusters. | C spec |
| L6 | **AMENDED 2026-07-03 (user ruling 2026-07-02):** the product is **Bernie** on every user-facing surface — plugin name, panel title, docs prose (baseline sweep shipped 2026-07-03). **k2000** remains the *internal* codename only (repo name, CMake project/targets, class prefixes) pending a possible later identifier pass — user chose baseline-only scope. *(Original: "Repo codename stays k2000".)* Tuning deferred to **v11**. | C spec; amended 2026-07-03 |
| L7 | **Spine filter is a `FilterModel` library** ([ADR-0011](../decisions/0011-selectable-spine-filter-library.md)) — append-only, stable-ID registry (Huggett v5.0 · Moog v5.2 · Oberheim SEM v5.3 · others later), selected per-Layer by an **automatable** `spine.filterModel`. **Live click-free switching** via a per-voice equal-power **crossfade** of two **heap-free, in-place** model instances. Param shape: a **common core** (cutoff/res/drive/output, always front-panel + mod-targetable) + **per-model namespaced banks** `spine.<modelId>.<param>` (the ADR-0008 idiom). | v5 |

| Q1 | **Stereo vs mono signal path.** **Resolved: full stereo throughout** — every block (sources, DSP graph, spine) is stereo from v5. *Deliberate modernization, confirmed against the hardware:* the real Summit is **mono through the dual filter + both drive stages, splitting to L/R only at the VCA** (Summit User Guide pp. 21–22 block diagram), and **K2088** VAST is **mono-per-layer + pan to stereo outputs** (K2088 Musician's Guide). We knowingly depart from that to be stereo end-to-end, accepting the ~2× per-voice filter cost (gated by Q11/Q12). | Touches every block, filter, voice mix, CPU. | v5 | 🟢 |
| Q2 | **Voice-count target.** ~~Resolved: 256~~ **Re-resolved 2026-07-02 with measured numbers (Q23): target 64–128 voices** (full stereo). | Q12 | **Filter oversampling policy at 256 voices.** **Direction resolved (constrained by L7):** the live-switch in-place/heap-free rule **forbids a per-voice `juce::dsp::Oversampling` object**. | Q14 | **Filter mode set exposed at v5. | Q15 | **White-box OTA vs gray-box tanh.** **v5.0 shipped gray-box** — three **asymmetric** tanh-class nonlinear stages (pre-drive input drive, resonance-loop saturator, post-drive), ADAA'd; see [nonlinear-filter-modeling.md](nonlinear-filter-modeling.md). | Q16 | **Filter promotion + preset migration. A migration test guards it. | Preset stability across the v5 promotion. | v5 | 🟢 |
| Q17 | **Crossfade duration + selector-automation debounce.** How long is the equal-power model-switch fade, and how do we bound cost when the (automatable) `spine.filterModel` is stepped rapidly (debounce / ignore no-op reselect / cap concurrent fades)? **Plan-1 findings (latent, currently unreachable with one model):** (a) `Layer::updateParameters` rebuilds `spineModel_` on id change, which would **dangle every Voice's `SpineFilterSlot::model_`/`state_`** — the hot-swap impl MUST re-bind/retire per-voice state (crossfade old→new) **before** freeing the old model, never free-and-forget; (b) `VoiceManager` voice-steal across layers does **not** re-prepare the spine, so it's only safe while all layers share model 0 — cross-layer reassignment must re-prepare/re-bind the spine once layers can hold different models. **Resolved (v5.1):** per-voice equal-power crossfade (`gOld=cos(p·π/2)`, `gNew=sin(p·π/2)`), ~25 ms default via automatable `spine.modelFadeMs` (2–100 ms), coalesce depth-1 (at most one fade in flight; newest pending wins). Q17b (cross-layer voice-steal) closed by note-start snap-bind (`SpineFilterSlot::bind`) — fresh/stolen voice reconstructs in-place state for its layer's model with no fade. | Live-switch cost + click-free guarantee. | v5 | 🟢 |
| Q18 | **Per-voice slot size budget governance.** The two in-place model slots are sized to the **largest** `FilterModel`. As models are appended (Oberheim+), what is the size cap, and what is the process when a new model would exceed it (bump size + migrate vs reject)? **Resolved (v5.1):** `kMaxSpineStateBytes=512` (Huggett measures ~176 B); a compile-time `static_assert` in `SpineFilterSlot` fires if a new model's `stateSize()` exceeds the cap — bump the constant or slim the model, never silently overflow. | Keeps L7's heap-free invariant as the library grows. | v5 (set) / ongoing | 🟢 |
| Q19 | **Live model-selector automation semantics.** `spine.filterModel` is a stepped choice automated live — confirm stepped (not interpolated) VST3 semantics and how the host/automation lane represents it; is the selector itself a mod-matrix destination? | Automation correctness for the selector. | v5 | 🔴 |
| Q20 | **Filter-validation harness B4 (phase / group delay) IR time-alignment.** The filter-characterization harness's `EssResponse` computes phase + group delay from the deconvolved impulse response, but the IR sits at the linear-convolution centre (~N samples), whose bulk latency wraps the phase faster than the probe grid can unwrap — so `phaseRad`/`groupDelaySec` are currently **descriptive-only / unreliable** (a synthetic 30-sample delay reads group delay ~9× off; surfaced by the `EssResponse magnitude is delay-invariant` test, 2026-06-30). Magnitude (B1), the method-agreement gate, and all spec/self-golden gates are magnitude-only and **unaffected** (magnitude is provably delay-invariant). **Fix:** time-align the deconvolved IR (shift sys + ref to a common reference origin, handling the band-limited pre-ringing) before the transfer function, then assert a calibrated pure delay reads `D/sr`. | B4 is the only battery whose output is not yet trustworthy; non-gating, so non-blocking for sub-project #1 merge. | filter-validation #1 follow-up | 🔴 |

| Q27 | **Huggett resonance loop is EXPANSIVE — defect or character?** SP-B Phase 0 ([large-signal read](../reviews/2026-07-02-huggett-large-signal-read.md), 2026-07-02) measured gain RISING with input level at the resonant peak (res 0.9: +61 dB small-signal → +86 dB at −6 dBFS in; output peak +89 dBFS; THD+N up to +5 dB), while Moog textbook-compresses (+5.8 → −3.3 dB, out ≤ −9 dBFS) — a ~98 dB output disparity at musical input. Suspect: the asymmetric `satRes`/`fbExtra` delta in `NlSvfCell::step()` rectifying feedback → effective damping falls with amplitude. **RULED (user, 2026-07-02): DEFECT — fix authorized now.** User field report: at max resonance the sound *disappears completely then re-appears* (consistent with state blow-up → Inf/NaN mute → recovery), and only the default-ON safety limiter prevented hearing damage; verdict "unusable as-is". Scope of the authorized fix: **stability + bounded, monotonically compressive resonance** (defect remediation, Moog-style internal ceiling pattern); the *authentic compression law/voicing* still calibrates against the real Summit at SP-D. **FIXED same day** (`fix/huggett-bounded-resonance`): root cause = transposed operands in the satRes delta (positive BP feedback growing with amplitude) + no state bound; fix = operand swap (damping grows with saturation) + OTA-style soft state rails (±4), mirrored into NlSvf/NlSvfDrive .cmajor + regenerated (bit-exact equivalence kept). Post-fix res 0.9: gain@−6 dBFS = +8 dB (was +86); voice at res 1.0/os8 peaks −1.5 dBFS (was +22). Small-signal + Moog byte-identical; goldens regenerated (144 Huggett fingerprint rows + gate peak proxy 33.8→25.0).
