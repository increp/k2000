# Franklin v1.1 — Stable Rendering, Test Info Cards, Run Provenance — Design

**Version:** 5.19 (artifact) · **Date:** 2026-07-05 · **Status:** Approved in brainstorm; awaiting written-spec review
**Parent:** `2026-07-03-franklin-dashboard-design.md` (v5.15, shipped as PR #14 / `e9e41e6`)
**Mandate:** user 2026-07-05 — Franklin is the future harness for external VST/hardware measurement and the comparison home for ours-vs-theirs; **hold it to an instrument-grade maturity bar.** Nothing in v1.1 may block future `capture`/`comparison` run kinds.

---

## 1. Problems (user field report, first live session)

1. **Dropdowns die in ~2 s.** The tab re-renders its entire DOM every runs-poll tick; form values are restored but an open native `<select>` cannot survive element replacement — the browser closes it on every tick. Unacceptable for any interactive control.
2. **Tests don't explain themselves enough.** Required per-test info card: what the test is · its purpose · **what is compared to what** · what failing means · what succeeding means · **who ran it** (the user, Claude, or CI). The catalog has 3 of the 6 fields; provenance doesn't exist.

## 2. Scope

**In:** stable-DOM section rendering (bug fix, structural); run provenance (producer → schema → UI); catalog schema v2 (+`compares`, +`succeedingMeans`, full 291 backfill, drift rule extended); info-card UI in run detail (with provenance banner) + a searchable **Catalog** browser section; forward-compatible run-`kind` handling; charter maturity line; manual updates.

**Out:** any capture/comparison implementation (charter territory); catalog entries for chz points (still generated); DOM test harness (lifecycle still inspection-verified + live-smoked).

## 3. Stable-DOM rendering (fixes the dropdown bug)

- `#app`'s Franklin view becomes **five persistent section containers** created once at mount: `fr-active`, `fr-ci`, `fr-form`, `fr-catalog`, `fr-archive`. Pollers update **only their own container's** innerHTML.
- **The form section renders only when its data changes** (templates/stale payload hash differs) — in steady state it is never touched, so its dropdowns live indefinitely.
- **Interaction guard:** any section repaint is skipped while `document.activeElement` (or an open `<details>` ancestor under edit) is inside that section; skipped paints coalesce into the next tick after focus leaves.
- Detail drawer keeps its cached-when-finished behavior; its container is owned by the archive section and repainted only when its run's data changes.
- Renderers stay pure (per-section renderers replace the monolithic `renderRunsPage`); tests pin per-section output + a "form HTML identical across unchanged-data repaints" assertion.

## 4. Run provenance

- **Schema:** `start` events gain `"runner": "ci" | "claude" | "dashboard" | "terminal"` (string; unknown values tolerated and rendered as-is — maturity rule).
- **Detection (in `runlog::Writer::start`, C++):** priority order `BERNIE_RUNNER` env (explicit override, any string) → `GITHUB_ACTIONS` set → `"ci"` → `CLAUDECODE` set → `"claude"` → `"terminal"`.
- **Dashboard spawns:** `control.ts` `startRun` adds `BERNIE_RUNNER=dashboard` to the child env — a Start-button run is **the user's**, not Claude's (user ruling 2026-07-05).
- **Display strings:** ci → "CI" · claude → "Claude" · dashboard → "you (dashboard)" · terminal → "you (terminal)"; anything else → verbatim.
- Old runlogs without the field render "unknown" — no migration (archives are immutable).

## 5. Catalog schema v2 + info cards

- Entry gains two required fields: `"compares"` (what is measured against what: golden file, analytic reference, fixed threshold, cross-method agreement — name the actual reference) and `"succeedingMeans"` (what a green result tells the user, in product terms — not "the test passes").
- **Full backfill of all 291 entries** (same bar as v1: no filler, reviewer-audited accuracy). `version` bumps to 2; the server serves either version; the UI requires v2 fields only when present (graceful during the branch).
- **Drift rule extended:** `franklin-catalog` now also WARNs on entries missing any of the five prose fields (what/why/compares/succeedingMeans/deviationMeans) — new tests can't ship half-carded.
- **Info card (run detail):** each test row expands to the six answers — What · Purpose · Compares · On success · On failure · **Run by** (from the run's `runner`, shown per card and as a banner on the run header: "run by you (dashboard) · 2026-07-05 14:31 · Release · <gitSha>").
- **Catalog browser:** new `fr-catalog` section — search box filtering the 291 cards by key/file/prose substring; each card shows the six fields with "Run by" replaced by "Last result" (latest archived suite run's pass/fail for that key, when available).

## 6. Forward-compatibility (maturity floor for capture/comparison)

- `RunKind` widens to `"chz" | "suite" | (string & {})` semantics: parser and renderers treat unknown kinds generically (neutral badge, label passthrough) instead of dropping the run. One test pins a `"capture"`-kind fixture rendering.
- No other speculative structure — capture/comparison design happens in its own spec when SP-D arrives.

## 7. Testing & verification

- Node: per-section renderer tests; form-stability assertion; provenance display mapping incl. unknown value; catalog v2 validation + browser search filter (pure function); unknown-kind fixture; control.ts env-injection test (spawned child env carries BERNIE_RUNNER=dashboard).
- C++: RunLogTests gains runner-detection cases (env-forced: override, GITHUB_ACTIONS, CLAUDECODE, none) — suite count changes tracked in README/drift as usual.
- Live smoke before PR: dropdown stays open >10 s during active polling with a run in flight; a dashboard-started run shows "you (dashboard)"; a terminal-started suite shows "you (terminal)"; catalog browser finds a known card.
- Full gauntlet (suite, drift `--ci`, npm, bundle) green.
